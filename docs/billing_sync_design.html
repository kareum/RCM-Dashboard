데이터 동기화 설계서

프로젝트 Billing Dashboard  ·  작성일 2025-03-30  ·  방식 Full sync (전체 비교)  ·  대상 DB clinics · patients · billing

1. 개요

원서버는 별도 webhook 또는 push 기능을 제공하지 않습니다. 따라서 우리 시스템이 주기적으로 원서버 API를 호출하여 전체 데이터를 가져온 후 내부 DB와 비교 처리합니다.

동기화 원칙
· 원서버가 정본(source of truth) — 충돌 시 원서버 데이터를 우선
· 내부에서만 생성되는 데이터(Invoice, Payment 등)는 동기화 대상 아님
· 삭제는 실제 DELETE 하지 않고 is_active = false 처리
항목내용
동기화 방식Full sync — 전체 데이터 가져와서 내부 DB와 비교
자동 주기매일 03:00 AM (cron: 0 3 * * *)
수동 트리거설정 화면 → "Sync now" 버튼
대상 데이터clinics · patients · billing (CPT 포함)
변경 감지sync_hash (MD5/SHA256) 비교
이력 기록api_sync_log 테이블에 모든 실행 기록 저장

3. 케이스별 처리 규칙

케이스감지 조건처리비고
신규INSERT 원서버 external_id가 내부 DB에 없음 patients / clinics에 INSERT
assignments에 초기 소속 INSERT		 is_active = true
변경UPDATE sync_hash 불일치 변환본 테이블 UPDATE
raw 테이블에 신규 row 추가
last_synced_at, sync_hash 갱신		 raw는 append-only
삭제감지 내부 DB에는 있는데 원서버 응답에 없음 is_active = false 실제 DELETE 하지 않음
클리닉 이동 patients의 소속 clinic_id 변경 감지 ① 기존 assignments row: unassigned_at = today, is_current = false
② 신규 assignments row: INSERT (assigned_at = today, is_current = true)		 이동 이력 보존 필수
변경 없음 sync_hash 일치 last_synced_at만 갱신 DB write 최소화

4. 대상별 동기화 상세

4-1. Clinics

항목내용
API endpointGET /api/clinics
주기매일 03:00 AM
hash 대상 필드name, state, phone, address, contact_name, service_types
주의EIN, NPI, Taxonomy, POS code, EMR links는 내부에서만 관리 — 원서버 값으로 덮어쓰지 않음
CMS-1500 billing 관련 필드(EIN, NPI 등)는 Clinic Settings에서 수동 입력하는 내부 전용 필드입니다. 동기화 시 원서버 값으로 덮어쓰지 않도록 주의해야 합니다.

4-2. Patients

항목내용
API endpointGET /api/patients
주기매일 03:00 AM
hash 대상 필드name, date_of_birth, medicare_id, insurance_type, qmb_status, deductible_status, service_types, icd10_codes, clinic_id(소속)
주의billing_hold는 내부에서 수동 설정 가능 — 원서버 QMB 값이 바뀌어도 billing_hold는 덮어쓰지 않음
클리닉 이동원서버의 clinic_id가 바뀌면 patient_clinic_assignments 이력 처리 실행
클리닉 이동 감지 로직은 반드시 트랜잭션으로 처리해야 합니다. assignments UPDATE와 INSERT가 하나의 트랜잭션 안에서 실행되지 않으면 데이터 불일치가 발생합니다.

4-3. Billing data

항목내용
API endpointGET /api/billing?month=YYYY-MM
주기매일 03:00 AM
범위현재 월 + 이전 2개월 (재청구 케이스 대비)
hash 대상 필드patient_id, clinic_id, cpt_code, billed_amount, service_date, units
주의내부에서 생성된 Claim 상태(Pre-claim, Denied 등)는 동기화로 변경하지 않음

5. 핵심 처리 로직 (Pseudocode)

5-1. Full sync 공통 로직

// Full sync 공통 처리 흐름 async function fullSync(target, triggeredBy) { const logId = await createSyncLog(target, triggeredBy) const startTime = Date.now() try { // 1. 원서버 전체 데이터 가져오기 const remoteData = await fetchFromRemote(target) // 2. raw 저장 (항상 append) await saveRaw(target, remoteData) // 3. 내부 DB와 비교 const localData = await getLocalData(target) const remoteIds = new Set(remoteData.map(r => r.external_id)) const localIds = new Set(localData.map(r => r.external_id)) for (const item of remoteData) { const hash = computeHash(item) const local = localData.find(l => l.external_id === item.external_id) if (!local) { await insertNew(target, item, hash) // 신규 } else if (local.sync_hash !== hash) { await updateChanged(target, item, hash) // 변경 } else { await updateSyncedAt(target, item.id) // 변경 없음 } } // 4. 삭제 감지 (원서버에 없어진 항목) for (const local of localData) { if (!remoteIds.has(local.external_id)) { await deactivate(target, local.id) // is_active = false } } await updateSyncLog(logId, 'success', Date.now() - startTime) } catch (err) { await updateSyncLog(logId, 'failed', Date.now() - startTime, err.message) throw err } }

5-2. 환자 클리닉 이동 감지

// 환자 클리닉 이동 처리 (트랜잭션 필수) async function handleClinicMove(patientId, newClinicId, changedBy) { await db.transaction(async (trx) => { // 1. 현재 소속 종료 await trx('patient_clinic_assignments') .where({ patient_id: patientId, is_current: true }) .update({ unassigned_at: today(), is_current: false, changed_by: changedBy, changed_at: now() }) // 2. 새 소속 등록 await trx('patient_clinic_assignments').insert({ patient_id: patientId, clinic_id: newClinicId, assigned_at: today(), is_current: true, changed_by: changedBy, changed_at: now() }) }) // 트랜잭션 종료 — 둘 다 성공하거나 둘 다 롤백 }

7. 에러 처리 및 모니터링

상황처리알림
원서버 API 타임아웃 3회 재시도 (30초 간격), 실패 시 api_sync_log에 error 기록 notifications 테이블 → 담당자 알림
부분 실패 (일부 레코드 오류) 오류 레코드 skip 후 나머지 처리 계속, error_detail에 기록 sync 완료 후 실패 건수 알림
트랜잭션 실패 (클리닉 이동) 전체 롤백, 해당 환자 skip 후 다음 처리 즉시 알림 (클리닉 이동 실패는 데이터 정합성 문제)
배치 자체 실행 실패 api_sync_log에 failed 기록 다음 날 배치 전 재시도 안 함 — 수동 실행 필요
